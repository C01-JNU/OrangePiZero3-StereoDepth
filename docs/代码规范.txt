# OrangePiZero3-StereoDepth 项目代码规范
# 最后更新：2026年1月17日
# 作者：C01-JNU

## 1. 命名规范

### 1.1 文件命名
- 头文件：使用小写字母，单词间用下划线分隔，例如：vulkan_pipeline.h
- 源文件：同上，例如：stereo_matching.cpp
- 类文件：类名与文件名保持一致，例如：CameraDriver 类对应 camera_driver.h

### 1.2 函数命名
- 使用小驼峰命名法，例如：calculateDisparity(), initVulkanDevice()
- 布尔函数以 is、has、can、should 开头，例如：isCameraConnected()

### 1.3 变量命名
- 局部变量：小驼峰命名法，例如：imageWidth
- 成员变量：前缀 m_ + 小驼峰，例如：m_deviceHandle
- 静态变量：前缀 s_ + 小驼峰，例如：s_instanceCount
- 常量：全大写，下划线分隔，例如：MAX_DISPARITY

### 1.4 类命名
- 使用大驼峰命名法，例如：StereoMatcher, CameraCalibrator

### 1.5 枚举和结构体
- 枚举类型：大驼峰，枚举值全大写
- 结构体：大驼峰，成员变量小驼峰

## 2. 代码格式

### 2.1 缩进
- 使用 4 个空格进行缩进，禁止使用 Tab

### 2.2 大括号
- 左大括号不换行
- 控制语句（if、for、while）必须使用大括号，即使只有一行

示例：
if (condition) {
    // code
}

### 2.3 空格使用
- 运算符两侧加空格
- 逗号后加空格
- 控制语句关键词后加空格

### 2.4 行长度
- 每行不超过 120 个字符

## 3. 头文件规范

### 3.1 包含保护
所有头文件必须使用 #pragma once

### 3.2 包含顺序
1. 相关头文件（对应源文件）
2. C 系统头文件
3. C++ 标准库头文件
4. 第三方库头文件
5. 项目内头文件

### 3.3 注释
- 每个头文件必须有文件头注释
- 每个函数必须有注释说明功能、参数、返回值
- 复杂逻辑需要行内注释

## 4. 错误处理

### 4.1 返回值
- 成功返回 true 或 0
- 失败返回 false 或错误码

### 4.2 异常
- 禁止使用 C++ 异常
- 使用返回值或错误码处理错误

## 5. 内存管理

### 5.1 资源获取即初始化（RAII）
- 所有资源管理必须使用 RAII 原则
- 使用智能指针管理动态内存

### 5.2 Vulkan 资源
- Vulkan 对象必须正确释放
- 使用作用域管理 Vulkan 资源生命周期

## 6. 日志和调试

### 6.1 日志级别
- ERROR: 严重错误
- WARN: 警告信息
- INFO: 一般信息

## 9. 性能规范

### 9.1 实时性
- 摄像头相关代码必须考虑实时性

最后更新：2026年2月8日
作者：C01-JNU
1. 文件命名
census.comp
cost.comp
aggregation.comp
wta.comp

2. 代码结构
2.1 头部注释
glsl
#version 450
// 着色器名称
// 功能描述
// 最后更新日期
// 作者
2.2 版本指令
必须使用 #version 450（Vulkan要求的最低版本）

2.3 工作组大小
glsl
layout(local_size_x = WORKGROUP_X, local_size_y = WORKGROUP_Y, local_size_z = 1) in;
使用配置生成的工作组大小宏

z维度固定为1

2.4 Uniform缓冲区
必须使用统一的 PipelineParams 结构体

绑定位置：binding = 0

必须使用 std140 布局

结构体必须与C++端完全匹配

2.5 缓冲区绑定
输入缓冲区：binding = 1, 2, 3...

输出缓冲区：binding = n, n+1...

类型：readonly buffer 或 writeonly buffer

3. 内存布局规则
3.1 std140对齐规则
标量（int, uint, float）：4字节对齐

向量（vec2, vec3, vec4）：按元素大小对齐

数组：每个元素按16字节对齐

结构体：按最大成员对齐

3.2 Uniform缓冲区结构体示例
glsl
layout(std140, binding = 0) uniform Parameters {
    // 第1组（16字节对齐）
    uint param1;    // 偏移0
    uint param2;    // 偏移4
    uint param3;    // 偏移8
    uint param4;    // 偏移12
    
    // 第2组（16字节对齐）
    float param5;   // 偏移16
    float param6;   // 偏移20
    uint param7;    // 偏移24
    uint padding;   // 偏移28（补齐）
} params;
4. 编程规范
4.1 变量命名
Uniform变量：params.xxx

局部变量：小驼峰命名法，例如：pixelIndex

常量：全大写，下划线分隔，例如：MAX_ITERATIONS

4.2 边界检查
必须进行边界检查：

glsl
if (x >= IMAGE_WIDTH || y >= IMAGE_HEIGHT) {
    return;
}
4.3 使用辅助宏
必须使用生成的辅助宏：

GET_PIXEL_INDEX(x, y)：获取像素索引

IS_VALID_PIXEL(x, y)：检查像素有效性

IS_FLAG_SET(flag)：检查标志位

4.4 错误处理
使用调试缓冲区输出错误信息

第一个像素用于记录验证信息

5. 性能优化
5.1 内存访问
使用局部变量缓存频繁访问的数据

避免全局内存的随机访问

使用向量化操作（vec4, vec8）

5.2 计算优化
减少分支语句

使用整数运算代替浮点运算（如果精度允许）

避免除法和模运算

5.3 Mali-G31特定优化
工作组大小：16x16（与硬件匹配）

避免使用64位数据类型

减少寄存器使用

6. 调试和验证
6.1 调试缓冲区
每个着色器应包含调试缓冲区：

glsl
layout(binding = N) writeonly buffer DebugBuffer {
    uint info[];
} debugBuffer;
6.2 验证机制
检查编译时常量与Uniform值的一致性

记录关键参数用于调试

输出验证结果（成功/失败）
